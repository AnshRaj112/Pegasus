Pegasus DuckDB reconciliation engine is still too slow for 10M–100M row CSV files.

Redesign the architecture for high-performance analytical reconciliation.

Critical Architecture Changes:

1. CSV Should Only Be Ingestion Format

- Convert CSV to Parquet immediately after ingestion
- Use Parquet as internal working format
- Add streaming CSV-to-Parquet conversion
- Add configurable row group sizes

2. Partitioned Reconciliation

- Partition datasets using:
  partition_id = hash(uid) % N
- Compare partitions independently
- Avoid giant global joins

3. DuckDB Optimization

- Enable parallel execution:
  PRAGMA threads=<cpu_cores>
- Use vectorized SQL operations
- Avoid Python row-by-row loops entirely
- Avoid full outer joins on gigantic datasets

4. Streaming + External Memory

- Use lazy scanning
- Avoid materializing full datasets
- Use spill-to-disk execution
- Keep memory bounded

5. UID Optimization

- Replace SHA256 with faster deterministic hashing
- Consider xxhash or murmurhash
- Avoid repeated hash generation

6. Mismatch Handling

- Write mismatches incrementally
- Avoid collecting mismatch arrays in memory
- Return only summary metadata to API

7. Query Optimization

- Push filters early
- Use column pruning
- Compare only required columns
- Add partition-level statistics

Required Modules:

- CSVIngestionPipeline
- ParquetConverter
- PartitionManager
- DuckDBReconciliationEngine
- MergeStreamComparator
- IncrementalMismatchWriter
- PerformanceMonitor

Technical Constraints:

- Avoid Python loops for reconciliation
- Use SQL/vectorized execution
- Use generators/iterators
- Add configurable chunk sizes
- Add temp-file cleanup
- Add memory + timing metrics

Focus ONLY on making 10M–100M row reconciliation fast and stable.
Do not add distributed Spark execution yet.
