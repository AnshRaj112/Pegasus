We are building a production-grade large-scale CSV/data reconciliation engine called Pegasus.

Goal:
Validate source and target datasets efficiently for files ranging from MBs to 100GB+ while being highly memory, RAM, CPU, and disk efficient.

Core Requirements:
- Stream-based processing only
- Never load entire files into memory
- Support multiple source and target files
- UID/key-based validation
- Generate mismatch/missing/extra reports
- Support sorted and unsorted datasets
- Handle enterprise-scale datasets

Architecture Requirements:

1. Streaming CSV Processing
- Use Node.js streams
- Use fast-csv or equivalent streaming parser
- Process rows incrementally
- Support backpressure handling

2. Sorted Merge Validation Engine
If source and target files are sorted by UID:
- Implement merge-join style validation
- Use pointer-based comparison
- O(1) or near-constant memory usage
- Compare rows sequentially

Logic:
- source.uid == target.uid → validate row
- source.uid < target.uid → missing in target
- source.uid > target.uid → extra in target

3. Sliding Window Matching
Implement configurable sliding window reconciliation:
- Example window size: 1000 rows
- Handle slightly out-of-order records
- Search nearby rows before declaring mismatch
- Useful for delayed/skewed ingestion pipelines

4. External Sorting for Unsorted Files
If datasets are not sorted:
- Implement external merge sort
- Split files into chunks
- Sort chunks individually
- Merge sorted chunks using k-way merge
- Avoid memory-heavy sorting

5. Hash Partition Validation
Implement partition-based validation:
- Partition rows by UID hash
- Example: hash(uid) % N
- Ensure same UID lands in same partition
- Validate partition-by-partition
- Designed for 100GB+ datasets

6. Validation Modes
System should automatically detect and choose:
- In-memory validation for small files
- Merge validation for sorted files
- External sort + merge for unsorted large files
- Hash partition validation for massive datasets

7. Report Generation
- Stream report writing
- Never store all mismatches in memory
- Generate:
  - missing rows
  - extra rows
  - mismatched fields
  - validation summary
- Support CSV/JSON reports

8. Row Hashing
Implement row fingerprinting:
- Generate normalized row hashes
- Compare hashes before field-level validation
- Avoid unnecessary deep comparisons

9. Multi-file Reconciliation
Support:
- multiple source files
- multiple target files
- many-to-one validation
- one-to-many validation

10. Fault Tolerance
- Add checkpointing
- Resume validation after crashes
- Track progress by byte offset or row number

11. Performance Requirements
Target:
- Process 100GB+ datasets
- Minimal RAM usage
- Sequential disk access preferred
- Avoid random disk I/O
- Parallel worker support later

12. Future Extensibility
Design architecture for:
- worker queues
- distributed processing
- GCS/S3 support
- parquet support
- compressed file streaming
- column-level validation rules

Implementation Instructions:
- Build modular architecture
- Separate parser, validator, comparator, partitioner, sorter, and reporter
- Use TypeScript
- Use interfaces and strategy patterns
- Avoid monolithic functions
- Add benchmarking and profiling utilities
- Add test datasets and stress tests
